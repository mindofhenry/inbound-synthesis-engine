## Header text spec

Single header block at the top, all caps, state appended after a pipe:

| State condition | Header `text` |
|---|---|
| `state === "HOT"` | `MQL ALERT - Inbound | HOT` |
| `state === "WARM"` | `MQL ALERT - Inbound | WARM` |

Implementation note: the scorer's `state` field is already uppercase
(`"HOT"` / `"WARM"`), so the header is a single concatenation:
`"MQL ALERT - Inbound | " + state`. No emoji on the header per Henry's
"max context, no decoration" intent (the prior `:rotating_light:`
context block is removed).

## Primary contact selection

**Locked decision (overrides current WF1 rule):**

1. Classify each contact by title:
   - **Tier 1** (decision_maker): title contains `VP`, `Chief`, `C-Suite`, `Director`, or `Head of`.
   - **Tier 2** (influencer): title contains `Lead`, `Manager`, or `Principal`.
   - **Tier 3** (end_user): everything else.
2. Within the highest-populated tier (1 first, else 2, else 3), pick the contact with the **highest first_party point sum** across that contact's `scored_signals`.
3. Tie within tier on point sum: most-recent `last_signal` (smallest `recency` days_ago) wins. (Defensive — unlikely to hit on the synthetic seed.)

Title regex (case-insensitive, word-bounded where relevant):

```js
function classifyTier(title) {
  if (!title) return 3;
  const t = title;  // raw casing preserved for "C-Suite" hyphen
  // Tier 1: VP, Vice President, Chief, C-Suite, Director, Head of (all word-boundary to avoid VPN, etc.)
  if (/\bVP\b/i.test(t)) return 1;
  if (/\b(Vice President|Chief|C-Suite|Director|Head of)\b/i.test(t)) return 1;
  // Tier 2: Lead, Manager, Principal (word-boundary)
  if (/\b(Lead|Manager|Principal)\b/i.test(t)) return 2;
  return 3;
}
```

**Per-contact first_party sum:** sum `scoredSignals.points` for signals
where `s.contact_id === contact.id && s.category === "first_party"`.
(Third-party signals like `bombora_surge` typically have `contact_id =
null` so this is a no-op for them, but the explicit category filter is
defensive against future signal types.)

**Best contact for orion-analytics under this rule:** Rachel Kim
(orion-001, VP Engineering) — tier 1, sole tier 1 contact, ~63 fp
points (pricing_page 25 + pricing_page 38 repeat). Sandy and Devon
fall to tier 2.

## Primary contact signal phrasing

Convert the primary contact's `scored_signals` (filtered to that
contact_id) into a comma-separated human-readable list. Repeat counts
collapse via `xN` suffix.

Algorithm:

```js
function phraseSignal(eventType, repeatCount) {
  const labels = {
    pricing_page: "Pricing page",
    competitor_compare: "Competitor compare",
    ceo_followup_content: "CEO follow-up content",
    whitepaper_form_fill: "Whitepaper download",
    webinar_attend: "Webinar attended",
    docs_page: "Docs page",
    longform_blog_read: "Long-form blog read"
  };
  const label = labels[eventType] || eventType;
  return repeatCount > 1 ? `${label} x${repeatCount}` : label;
}
```

Group by `event_type` first (max `repeat_count` from
`Transform to Scoring Payload` already encodes per-(type,contact)
repetition), then `.join(", ")`.

**Example for Rachel Kim:** `Pricing page x2, G2 compare`. The G2 view
is third-party (`category === "third_party"`); per Henry's "specific
signals they touched" framing, third-party events tied to the same
contact_id stay in the line — but `bombora_surge` (typically
contact_id = null) does not appear under any contact.

## Secondary contacts line

All `all_contacts` minus the primary, joined as
`"Name (Title), Name (Title)"`. Header text reads: `"N others active:"`
where `N = all_contacts.length - 1`.

**Example for orion-analytics:** `2 others active: Sandy Hallward
(Analytics Lead), Devon Cole (Ops Lead)`.

If there is exactly one secondary: `"1 other active: …"`. If zero
secondary contacts (i.e., `all_contacts.length <= 1`), **omit block 6
entirely** — the section is conditionally appended in the Slack node
body, not rendered with empty text.

## Synthesis blurb

1–2 sentence Claude-Haiku-generated rationale. Lives in its own Block
Kit `section` directly under the primary contact. Italics in mrkdwn
(`_..._`). Generated by a new `Synthesize Alert Blurb` HTTP Request
node placed between `Score Account + Pick Contact` and `Write
inbound_alerts Row`.

Constraints (enforced via prompt, not code — MIN-72 validator lands
later):

- Reference only the contacts, signals, and timestamps in the input payload. No invented dates or facts.
- Output 1–2 sentences. No preamble, no markdown, no list formatting.
- Connect the rep's strongest signal to the company_context pain or differentiator that maps to it.
- Do not use em-dashes.
- Do not write outbound copy. Do not propose specific messaging.

### Haiku request shape

- Model: `claude-haiku-4-5-20251001`
- `max_tokens`: 200 (enough for two sentences with headroom; cuts off runaway prose)
- `system`: see below
- `messages[0].content`: composed user message naming primary contact + their signals + a compact account context

### System prompt (verbatim — paste into the node)

```
You are the synthesis layer for a B2B GTM alert. A rep just received a Slack notification that an account has crossed the HOT or WARM threshold. Your job is to write one or two sentences explaining why the primary contact's signals matter, grounded in the company_context provided.

Hard rules:
- Reference only the contacts, signals, and timestamps present in the input payload. Do not invent dates, names, titles, signal types, or facts not in the payload.
- Output exactly one or two sentences. No preamble, no markdown, no list formatting, no headers, no quotes.
- Connect the rep's strongest signal (the highest-points event the primary contact touched) to the company_context pain or differentiator that maps to it.
- Do not use em-dashes. Use periods or commas.
- Do not write outbound copy. Do not propose messaging, openers, or what to say to the prospect.
- Do not restate the score, the state (HOT/WARM), the contact's name, or the contact's title — those already appear elsewhere in the alert.
- If the company_context names the buying stage implied by the signals (e.g., evaluation, comparison, late-stage), say so. Otherwise stay descriptive.
```

### User message template

The HTTP Request node builds this dynamically from `$json` (output of
the scorer) plus the `company_context` Set node. Pseudocode:

```js
const ctx = $node["Set Company Context"].json.company_context;
const primary = $json.best_contact;
const primarySignals = $json.scored_signals
  .filter(s => s.contact_id === primary.id)
  .sort((a, b) => b.points - a.points);  // strongest first
const allContactSigCounts = {};
for (const s of $json.scored_signals) {
  if (!s.contact_id) continue;
  allContactSigCounts[s.contact_id] = (allContactSigCounts[s.contact_id] || 0) + 1;
}

const userMsg =
  "Company context (Blast Radius vendor profile):\n" + ctx + "\n\n" +
  "Account: " + $json.account.name + "\n" +
  "Account state: " + $json.state + " (score " + $json.scoring.final_score + "/100)\n" +
  "Velocity: " + $json.scoring.velocity_recent_signals + " signals in 7 days, " + $json.scoring.distinct_contacts + " distinct contacts.\n\n" +
  "PRIMARY CONTACT:\n" +
  "- Name: " + primary.name + "\n" +
  "- Title: " + primary.title + "\n" +
  "- Signals (in order of point weight, all within last 30 days):\n" +
  primarySignals.map(s => "  * " + s.type + (s.repeat_count > 1 ? " (x" + s.repeat_count + ")" : "") + " — " + s.days_ago + " days ago, " + s.points + " pts, " + s.category).join("\n") + "\n\n" +
  "SECONDARY CONTACTS (named only, signal counts):\n" +
  $json.all_contacts.filter(c => c.id !== primary.id).map(c => "- " + c.name + " (" + c.title + "), " + (allContactSigCounts[c.id] || 0) + " signals").join("\n") + "\n\n" +
  "Write the 1–2 sentence synthesis now. Follow every rule in the system prompt.";
```

Wired into the n8n `httpRequest` node body as
`={{ JSON.stringify({ model: "claude-haiku-4-5-20251001", max_tokens: 200, system: "<paste verbatim>", messages: [{ role: "user", content: <expression that builds userMsg> }] }) }}`.

### Synthesis output extraction

Anthropic `/v1/messages` returns
`{ content: [{ type: "text", text: "..." }] }`. The downstream Slack
node references the blurb as
`$("Synthesize Alert Blurb").item.json.content[0].text`. Trim
whitespace; if the response is empty (rare on Haiku), fall back to a
literal `"Synthesis unavailable."` so Slack still renders.
